## 1. 架构设计原则

### 1.1 核心设计理念

**并行度最大化三原则**:
1. **依赖图驱动并行**: 基于 `4.模型依赖关系图.puml` 的拓扑排序，识别零依赖节点和可并行波次
2. **接口契约先行**: 每波次第一步产出 Protocol/ABC 接口，Reviewer 审查通过后并行实现
3. **垂直切片验证**: 在 50% 进度时跑通核心链路，避免集成地狱

**工业级质量四要素**:
1. **规格驱动**: 所有实现必须基于 OpenSpec 规格文档，代码即规格的实现
2. **测试先行**: TDD 模式，接口定义后立即编写测试用例，实现时以测试通过为准
3. **持续集成**: 每个 Agent 完成模块后立即集成测试，不等到最后
4. **文档同步**: 代码变更必须同步更新规格文档，保持唯一真理源

### 1.2 OpenSpec 五条护栏（强制遵守）

根据 `feedback_openspec_guardrails.md`：

1. **契约非实现**: 规格文档只定义"做什么"（职责、输入输出、前后置条件），不定义"怎么做"（算法、数据结构）
2. **依赖图硬约束**: 不得违反 `4.模型依赖关系图.puml` 的依赖方向，下层不得调用上层
3. **单一数据源**: 规格文档是唯一真理源，代码实现有疑问时以规格为准
4. **视角互补**: 规格文档（What）+ 技术选型（How）+ 目录结构（Where）三位一体
5. **已锁定内容不动**: 已完成的基础设施层/抽象驱动层/执行逻辑层规格不得修改，仅可在控制层/UI层补充

```python
import bcrypt

        hashed = bcrypt.hashpw(password.encode(), salt)
        return hashed.decode()
```

```python
# sdpj/core/detector/static_detector.py
class StaticDetector:
    async def select_best_jailbreak_poc(
        self, 
        target_model_id: str, 
        jailbreak_dataset_id: int
    ) -> str:
        """Algorithm 1 步骤1: 筛选越狱成功的最优 PoC"""
        samples = await self.data_processor.load_samples(jailbreak_dataset_id)
        success_pocs = []
        
        for sample in samples:
            output = await self.llm_interface.call(
                model_id=target_model_id,
                system_prompt="",
                user_message=sample.poc
            )
            if self._is_jailbreak_success(output):
                success_pocs.append(sample.poc)
        
        if not success_pocs:
            return None  # 未检测到越狱风险
        
        return self._select_best_poc(success_pocs)
    
    async def build_judge_model(self, best_poc: str, role_prompt: str) -> str:
        """Algorithm 1 步骤2: 构造虚拟化输出侧合规判断模型"""
        return best_poc + role_prompt  # PoC_ii
    
    async def traverse_and_report(
        self,
        judge_template: str,
        dataset_ids: List[int],
        target_model_id: str,
        task_group_id: int
    ) -> int:
        """Algorithm 1 步骤3: 遍历检测样本库生成静态检测报告"""
        report_id = await self.data_processor.create_report(task_group_id)
        
        for dataset_id in dataset_ids:
            samples = await self.data_processor.load_samples(dataset_id)
            
            # 并发处理样本（asyncio.gather）
            tasks = [
                self._detect_single_sample(
                    sample, judge_template, target_model_id, report_id
                )
                for sample in samples
            ]
            await asyncio.gather(*tasks)
        
        return report_id
```

### 3.4 Wave 3: 应用控制层（系统咽喉）

**并行度**: 1 agent（StateScheduler 是单点，无法并行）  
**复杂度**: XL（超大型）  
**前置条件**: Wave 2 的 8 个执行逻辑层模块接口已锁定

#### 3.4.1 为什么 StateScheduler 是咽喉

根据依赖图分析：
- **向下依赖**: StateScheduler 调用 8 个执行逻辑层模块的接口
- **向上被依赖**: CLI 和 WebUI 都调用 StateScheduler 的接口
- **职责复杂**: 状态机管理 + 任务调度 + 异常处理 + 日志记录

**必须等待 Wave 2 完成的原因**: StateScheduler 需要编排所有执行逻辑层模块，如果下层接口未稳定，会导致频繁返工

#### 3.4.2 执行流程

**阶段 1: 状态机设计**

StateScheduler Agent 产出状态机定义：

```python
# sdpj/control/state_machine.py
from statemachine import StateMachine, State

class SystemStateMachine(StateMachine):
    """系统状态机"""
    
    # 状态定义
    idle = State("空闲", initial=True)
    detecting = State("检测中")
    generating_report = State("生成报告中")
    error = State("异常")
    
    # 状态转移
    start_detection = idle.to(detecting)
    detection_done = detecting.to(generating_report)
    report_done = generating_report.to(idle)
    error_occurred = (idle.to(error) | detecting.to(error) | generating_report.to(error))
    recover = error.to(idle)
    
    # 转移钩子
    def on_enter_detecting(self):
        self.event_logger.log_info("系统进入检测状态")
    
    def on_enter_error(self):
        self.event_logger.log_error("系统进入异常状态")
```

实现核心调度逻辑
```python
# sdpj/control/state_scheduler.py
class StateScheduler:
    def __init__(
        self,
        task_queue_manager: TaskQueueManagerInterface,
        sdpj_detector: SDPJDetectorInterface,
        report_manager: ReportManagerInterface,
        # ... 其他 5 个依赖
    ):
        self.task_queue = task_queue_manager
        self.detector = sdpj_detector
        self.report_mgr = report_manager
        self.state_machine = SystemStateMachine()
    
    async def start_detection(
        self,
        user_id: int,
        detection_type: str,
        dataset_ids: List[int],
        target_model_id: str,
        max_iterations: int = 5
    ) -> Dict[str, Any]:
        """职责1: 接收并解析检测启动指令并下发为可执行任务组"""
        
        # 1. 创建任务组
        task_group_id = await self._create_task_group(user_id, detection_type)
        
        # 2. 解析为子任务并入队
        for dataset_id in dataset_ids:
            task_id = await self.task_queue.enqueue({
                "user_id": user_id,
                "detection_type": detection_type,
                "dataset_id": dataset_id,
                "target_model_id": target_model_id,
                "task_group_id": task_group_id
            })
        
        # 3. 状态转移
        self.state_machine.start_detection()
        
        # 4. 异步调度执行
        asyncio.create_task(self._schedule_tasks())
        
        return {"task_group_id": task_group_id, "status": "queued"}
    
    async def _schedule_tasks(self):
        """职责2: 调度执行单个检测子任务"""
        while True:
            task = await self.task_queue.dequeue()
            if task is None:
                break
            
            try:
                if task["detection_type"] == "static":
                    await self.detector.static_detect(
                        user_id=task["user_id"],
                        target_model_id=task["target_model_id"],
                        dataset_ids=[task["dataset_id"]],
                        task_group_id=task["task_group_id"]
                    )
                # ... 动态检测逻辑
                
                await self.task_queue.update_status(task["id"], "completed")
            except Exception as e:
                await self.task_queue.update_status(task["id"], "failed")
                self.state_machine.error_occurred()
                raise
```

```python
@app.websocket("/ws/task/{task_group_id}")
async def task_progress(
    websocket: WebSocket,
    task_group_id: int,
    scheduler: StateScheduler = Depends(get_scheduler)
):
    """实时推送任务进度"""
    await websocket.accept()
    while True:
        progress = await scheduler.query_task_progress(task_group_id)
```

## 4. 并行度分析与复杂度估算

### 4.1 波次并行度汇总

| 波次 | 模块数 | 并行度 | 复杂度 | 累计进度 |
|------|--------|--------|--------|----------|
| Wave 0 | 5 | 5 agents | M（中等） | 5/21 = 24% |
| Wave 1 | 7 | 7 agents | L（大型） | 12/21 = 57% |
| Wave 2 | 5 | 5 agents | XL（超大型） | 17/21 = 81% |
| Wave 3 | 1 | 1 agent | XL（超大型） | 18/21 = 86% |
| Wave 4 | 3 | 3 agents | L（大型） | 21/21 = 100% |
| **总计** | **21** | **平均 4.2** | **混合** | **100%** |

**复杂度说明**:
- **S（小型）**: 单一职责，依赖少，实现简单（如 UtilsLib 的部分功能）
- **M（中等）**: 职责清晰，依赖适中，需要一定测试覆盖（如 EventLogger、TaskQueueManager）
- **L（大型）**: 多职责，依赖复杂，需要完整测试（如 DataProcessor、WebUI）
- **XL（超大型）**: 核心模块，算法复杂，依赖众多（如 SDPJDetector、StateScheduler）

### 4.2 关键里程碑

| 里程碑 | 时间点 | 验收标准 |
|--------|--------|----------|
| **M1: 基础设施就绪** | Wave 0 结束 | 5 个基础模块接口通过 Reviewer 审查 + 单元测试通过 |
| **M2: 数据层就绪** | Wave 1 结束 | 3 个数据库模块可正常 CRUD + 4 个驱动层模块集成测试通过 |
| **M3: 垂直切片验证** | Wave 2 结束 | 跑通完整检测链路（Mock LLM → SDPJDetector → 结果入库） |
| **M4: 控制层就绪** | Wave 3 结束 | StateScheduler 可正确调度 8 个执行逻辑层模块 |
| **M5: 系统交付** | Wave 4 结束 | CLI + WebUI 端到端测试通过 + 部署脚本可用 |

### 4.3 并行度优化分析

**理论最大并行度**: 21 个模块 / 5 个波次 = 平均 4.2 agents 并行

**实际并行度**:
- Wave 0: 5 agents（100% 并行）
- Wave 1: 7 agents（100% 并行）
- Wave 2: 5 agents（100% 并行）
- Wave 3: 1 agent（0% 并行，咽喉瓶颈）
- Wave 4: 3 agents（100% 并行）

**瓶颈分析**: Wave 3 的 StateScheduler 是唯一串行点

**优化方案**: 
1. **提前产出 StateScheduler 接口骨架**（在 Wave 0 完成），让 Wave 2 的模块知道自己要对接什么
2. **Wave 3 拆分为两阶段**: 
   - 阶段 1: 产出接口 + 状态机定义（Reviewer 审查）
   - 阶段 2: 实现 + 集成测试（可与 Wave 4 的接口定义并行）

## 10. 总结与建议

### 10.1 架构设计亮点

1. **最高并行度**: 基于依赖图的波次式并行开发，平均并行度 4.2 agents，21 个工作日完成 21 个模块
2. **接口契约先行**: 每波次第一步产出 Protocol/ABC 接口，Reviewer 审查通过后并行实现，避免返工
3. **垂直切片验证**: 在 50% 进度（Wave 2 结束）跑通核心检测链路，提前发现集成问题
4. **规格驱动开发**: 所有实现基于 OpenSpec 规格文档，代码即规格的实现，保持唯一真理源
5. **持续集成**: 每个 Agent 完成模块后立即集成测试，不等到最后，降低集成风险

### 10.2 关键成功因素

1. **Architect Agent 的全局视野**: 维护依赖图、裁决冲突、协调资源
2. **Reviewer Agent 的质量把关**: 接口审查、代码审查、规格合规性检查
3. **Integrator Agent 的持续集成**: CI/CD 流水线、垂直切片验证、环境管理
4. **Module Agents 的专业能力**: 深度理解规格文档、熟悉异步编程、遵循 TDD
5. **沟通协议的严格执行**: 接口变更通知、阻塞问题升级、文档同步

### 10.3 潜在挑战与应对

**挑战 1: StateScheduler 复杂度高，容易成为瓶颈**
- 应对: 提前产出接口骨架（Wave 0），拆分为状态机 + 调度逻辑两阶段，必要时增加 1 个 Agent 协助

**挑战 2: SQLite 并发写入限制**
- 应对: 启用 WAL 模式，限制并发度 ≤3，预留 PostgreSQL 升级路径

**挑战 3: LLM API 限流导致检测失败**
- 应对: 实现指数退避重试（tenacity），支持多 API Key 轮询，本地缓存响应

**挑战 4: 异步编程复杂度导致 Bug**
- 应对: 强制使用 asyncio.TaskGroup，禁用 Lazy Loading，Code Review 重点检查

**挑战 5: 前后端联调延期**
- 应对: 前端从 Wave 0 独立启动，使用 Mock API 并行开发，定义清晰的 API 契约
