StateScheduler / 系统状态管理及调度控制模块

具体职责:

# 检测调度
1. 接收并解析检测启动指令并下发为可执行任务组
   - 输入:用户ID、检测类型(静态/动态)、用户选用的检测数据集集合或私有检测配置标识、目标被测大模型标识、动态算法所需的最大迭代次数等参数
   - 输出:本次检测对应的检测任务组标识、各子任务及其队列任务标识
   - 后置条件:用户的"勾选多个安全风险类型"等高层意图被解析并转换为"一个检测任务组",其下每个数据集对应一个独立子任务,任务组内各子任务已加入检测任务队列
   - 触发场景:用户启动 SDPJ 静态/动态自检测算法、调度新的检测任务(对应 1.spec.md 功能 1.1 / 1.2 / 1.4)
   - 调用下层:PrivateConfigManager 的「读取用户私有检测配置」加载用户已选检测配置;TaskQueueManager 的「接收新检测任务并加入队列尾部」逐子任务入队;EventLogger 的「记录用户操作日志」与「记录系统运行日志」
   - 不负责的边界:不做检测算法的具体执行逻辑(由 SDPJDetector 完成);不做用户私有检测配置内容的持久化(由 PrivateConfigManager 完成);不做用户对所选私有资源的访问权判定(在调用前转交 DACManager 完成)

2. 调度执行单个检测子任务
   - 输入:队列任务标识、对应子任务的业务参数(被测大模型ID、检测类型、目标数据集标识等)
   - 输出:子任务执行进度与结束状态
   - 后置条件:子任务从队列中出队、状态由「等待中」转为「进行中」直至「已完成」/「异常中断」;检测明细已由 SDPJDetector 经下层落盘
   - 触发场景:任务队列中存在待执行子任务时由 StateScheduler 主动取出并驱动执行(对应 1.spec.md 功能 1.1 / 1.2 / 1.4)
   - 调用下层:TaskQueueManager 的「按 FIFO 顺序取出下一个待执行任务」与「更新单个任务的执行状态」;SDPJDetector 的「静态自检测算法执行」相关能力(具体为「筛选越狱成功的最优 PoC」「构造虚拟化输出侧合规判断模型」「遍历检测样本库生成静态检测报告」)或「动态自检测算法执行」相关能力(具体为「构造虚拟化动态变异红队攻击模型」「迭代变异并生成动态检测报告」);EventLogger 的「记录系统运行日志」(任务进度推进事件)
   - 不负责的边界:不做检测样本/PoC 的业务语义审查(由 SDPJDetector / DataProcessor 完成);不直接访问检测结果数据库(由 SDPJDetector 经 DataProcessor 完成)

3. 并发调度多个检测子任务
   - 输入:目标并发度上限
   - 输出:本批次并发执行的子任务集合及其结束状态
   - 后置条件:被取出的多个子任务并发推进,状态分别更新;执行结果按子任务独立落盘
   - 触发场景:用户连续下发多个检测任务时由 StateScheduler 协调并发执行(对应 1.spec.md 功能 1.4)
   - 调用下层:TaskQueueManager 的「按 FIFO 顺序并发取出多个待执行任务」与「更新单个任务的执行状态」;SDPJDetector 的检测算法执行相关能力(同职责2);EventLogger 的「记录系统运行日志」
   - 不负责的边界:不做实际的检测算法并发实现细节(由 SDPJDetector 内部以单线程异步实现);不做跨任务的依赖/优先级编排(队列仅 FIFO,由 TaskQueueManager 边界声明)

4. 查询检测任务执行进度
   - 输入:队列任务标识或无(查询整体队列视图)
   - 输出:单个任务的当前状态或当前队列中全部任务的状态视图
   - 触发场景:用户查看系统当前检测任务的排队与进度状况(对应 1.spec.md 功能 1.4 / 2.1)
   - 调用下层:TaskQueueManager 的「查询单个任务的执行状态」或「查询队列整体视图」
   - 不负责的边界:不做进度信息的可视化渲染(由 UI 层完成)

# 报告管理调度
5. 生成检测报告
   - 输入:已完成的检测任务组ID、检测类型(静态/动态)
   - 输出:对应检测任务组的标准化检测报告
   - 触发场景:某检测任务组所有子任务结束后由 StateScheduler 触发报告生成(对应 1.spec.md 功能 1.5)
   - 调用下层:静态检测对应 ReportManager 的「生成静态检测报告」,动态检测对应 ReportManager 的「生成动态检测报告」;EventLogger 的「记录用户操作日志」与「记录系统运行日志」
   - 不负责的边界:不做报告统计指标的具体计算(由 ReportManager 完成);不做合规判断(由 SDPJDetector 完成)

6. 查看与查询检测报告
   - 输入:任务组ID(查看单份)或过滤条件(列表查询时,如归属用户、被测大模型等)
   - 输出:单份检测报告的完整内容或匹配条件的报告元信息列表
   - 触发场景:用户浏览历史检测报告或查看某份报告详情(对应 1.spec.md 功能 1.6)
   - 调用下层:ReportManager 的「查看单份检测报告」或「查询检测报告列表」;EventLogger 的「记录用户操作日志」
   - 不负责的边界:不做报告内容的可视化渲染(由 UI 层完成);不做用户对报告的访问权判定(由 DACManager 在调用前完成)

7. 删除检测报告
   - 输入:任务组ID / 任务ID / 检测报告ID / 结果数据ID(按删除粒度任选其一)、调用者用户ID
   - 输出:删除结果
   - 后置条件:对应粒度的报告及其级联结果由下层清理
   - 触发场景:用户删除某份历史报告或精修报告内容(对应 1.spec.md 功能 1.6)
   - 调用下层:DACManager 的「判定用户对资源是否具备访问权」做前置访问权校验;ReportManager 的「删除检测报告」;EventLogger 的「记录用户操作日志」

8. 导出检测报告与可视化数据准备
   - 输入:任务组ID、目标导出文件格式或可视化图表准备意图
   - 输出:可供下载的检测报告文件内容或按图表类型组织的可视化数据
   - 触发场景:用户下载历史检测报告或在 UI 层展示可视化图表(对应 1.spec.md 功能 1.5 / 1.6)
   - 调用下层:ReportManager 的「导出检测报告文件」或「准备可视化图表数据」;EventLogger 的「记录用户操作日志」
   - 不负责的边界:不做报告文件序列化的具体实现(由 ReportManager 经 DataProcessor 完成);不做图表 UI 渲染(由 UI 层完成);检测报告不在 1.spec.md 功能 2.1.8.2 列明的敏感字段范围内,不涉及 C-S 链路加密传输

# 系统状态与日志
9. 维护系统当前状态并向上层反馈进度与事件
   - 输入:本模块内部 FSM 的状态转移触发事件(用户指令到达、子任务进度推进、异常发生、报告生成完成等)
   - 输出:当前系统状态、向上层(CLI / WebUI)推送的状态变化与执行进度反馈
   - 后置条件:本模块持有"系统当前状态"在内存中;每一次状态转移产生对应的运行日志条目
   - 触发场景:任一指令执行过程中或检测任务推进时(对应 1.spec.md 功能 2.1 / 2.1.7.1)
   - 调用下层:EventLogger 的「记录系统运行日志」
   - 不负责的边界:不持有任何业务数据(业务数据由各下层 DB 模块持有);不做状态信息的 UI 渲染(由 UI 层完成);不做状态机状态定义/转移规则的具体实现内容(属于实现细节,由「技术细节」段约束)

10. 在状态转移与业务执行中按事件类别记录日志
    - 输入:操作者用户ID(若涉及用户操作)、事件类型(操作/运行/错误)、事件上下文(操作对象、关键参数、错误描述等)
    - 输出:日志条目标识或记录结果
    - 触发场景:用户触发任一业务操作时(操作日志);系统状态转移、检测进度推进等运行期事件(运行日志);业务执行过程中出现的异常/失败事件(错误日志)。对应 1.spec.md 功能 2.1.7.2 / 2.1.7.3
    - 调用下层:EventLogger 的「记录用户操作日志」「记录系统运行日志」「记录系统错误日志」
    - 不负责的边界:不做日志的查询展示(其查询调用见职责11);不做敏感信息脱敏规则(由本模块在传入日志参数前完成,具体规则属配置层)
    - **实现说明**:本职责为内部横切行为,由各业务方法在执行过程中直接调用 EventLogger,不在 StateSchedulerInterface 上暴露独立方法

11. 查询系统日志
    - 输入:过滤条件(日志类别、时间范围、来源模块、用户ID 等,可选)
    - 输出:匹配条件的日志条目列表
    - 触发场景:运维或用户查看系统运行状况(对应 1.spec.md 功能 2.1.7.2)
    - 调用下层:EventLogger 的「按条件查询日志」
    - 不负责的边界:不做日志结果的 UI 渲染(由 UI 层完成)

12. 处理业务执行过程中的异常并按状态转移规则恢复
    - 输入:下层模块向上反馈的不可恢复错误或本模块在业务执行中识别到的异常事件
    - 输出:转入相应异常状态、必要时中断或重置当前任务/会话、向上层反馈错误信息
    - 后置条件:对应任务的状态被更新(如「异常中断」),错误事件被记录,当前 FSM 状态转入恢复后的目标状态
    - 触发场景:大模型 API 调用不可恢复错误、数据库访问错误、权限校验失败、文件格式校验失败等业务执行过程中出现的异常事件(对应 1.spec.md 功能 2.1.7.3)
    - 调用下层:TaskQueueManager 的「更新单个任务的执行状态」(置为「异常中断」);EventLogger 的「记录系统错误日志」
    - 不负责的边界:不做下层错误的具体分类与重试/退避决策(分别由 LLMAdapterLib 与 LLMService 完成,且属跨层,本模块仅消费上抛错误);不做异常的 UI 提示渲染(由 UI 层完成)

# 用户账号调度
13. 调度用户注册与登录
    - 输入:账号、明文密码
    - 输出:注册结果(附用户ID)或登录结果(附用户ID 与登录会话状态)
    - 后置条件:登录成功时本模块感知"当前登录用户"以驱动后续指令的归属判定
    - 触发场景:用户通过 UI 层发起注册或登录(对应 1.spec.md 功能 3.1.1.1)
    - 调用下层:AccountManager 的「用户注册」或「用户登录」;EventLogger 的「记录用户操作日志」
    - 不负责的边界:不做账号格式/密码强度的具体校验规则(由 AccountManager 在前置校验时完成);不做密码哈希(由 UserCenter 完成)

14. 调度账号管理操作
    - 输入:操作类型(密码修改/用户名修改/账号切换/登出/注销/查询用户资料/列出用户拥有的受控资源)及其所需参数
    - 输出:对应操作的结果
    - 触发场景:用户在账号管理界面操作个人账号(对应 1.spec.md 功能 3.1.1.2)
    - 调用下层:按操作类型分别调用 AccountManager 的「修改指定用户密码」「修改指定用户用户名」「账号切换」「用户登出」「用户注销」「查询指定用户资料」「列出指定用户拥有的受控资源」;EventLogger 的「记录用户操作日志」
    - 安全说明:涉及用户身份的操作（查询资料、修改密码/用户名、列出资源）所需的 user_id 由 WebUI 路由从 request.state.user_id（认证中间件写入）注入到 params，不依赖 AccountManager 内部单例状态，避免多用户并发时身份混淆
    - 不负责的边界:不持久化登录会话(登录会话由 AccountManager 在内存中维护)

# 权限授权调度
15. 调度自主访问控制相关操作
    - 输入:操作类型(授予读权限/移除读权限/查询资源授权清单)及其所需参数(资源ID、被授权用户ID、访问控制项ID 等)、调用者用户ID
    - 输出:对应操作的结果(新创建访问控制项ID、移除结果、授权清单等)
    - 触发场景:资源拥有者管理其私有资源的访问授权(对应 1.spec.md 功能 3.1.2)
    - 调用下层:DACManager 的「授予其他用户对私有资源的读权限」「移除已授予的读权限」「查询某资源的授权清单」之一;EventLogger 的「记录用户操作日志」
    - 不负责的边界:不做受控资源内容管理(由 PrivateConfigManager 完成);不做权限委托或被授权用户再授权(由 DACManager 在边界声明)

16. 在使用私有资源前的访问权前置校验
    - 输入:资源ID、当前调用者用户ID
    - 输出:是否具备访问权的判定结果
    - 触发场景:用户对私有资源(私有检测配置/私有数据集/私有大模型适配器/检测报告等)发起读取/使用/导出/删除请求前的前置校验(对应 1.spec.md 功能 3.1.2)
    - 调用下层:DACManager 的「判定用户对资源是否具备访问权」
    - 不负责的边界:不做权限规则的实现(规则在 DACManager 内部);不替代私有资源业务模块自身的"调用者必须为拥有者"等强约束(由 DACManager / PrivateConfigManager 自检)

# 私有资源调度
17. 调度用户私有检测配置的全生命周期管理
    - 输入:操作类型(创建/读取/修改/删除/列出/导入/导出)及其所需参数
    - 输出:对应操作的结果(新建配置ID、配置内容、修改结果、删除结果、配置清单、导出文件、导入后新配置ID 等)
    - 触发场景:用户在私有配置管理界面操作私有检测配置(对应 1.spec.md 功能 3.1.1.2);系统在启动检测前加载用户选择的检测配置(对应 1.spec.md 功能 1.1 / 1.2 / 1.3)
    - 调用下层:涉及私有资源的非创建操作(读取/修改/删除/导出/导入)前,调用 DACManager 的「判定用户对资源是否具备访问权」做前置校验;按操作类型分别调用 PrivateConfigManager 的「创建用户私有检测配置」「读取用户私有检测配置」「修改用户私有检测配置」「删除用户私有检测配置」「列出用户的私有检测配置清单」「导入用户私有检测配置」「导出用户私有检测配置」;EventLogger 的「记录用户操作日志」
    - 不负责的边界:不做配置文件格式校验的具体实现(由 PrivateConfigManager 经 DataProcessor 完成);不做私有配置内容的存储(由 PrivateConfigManager 经 UserCenter 完成)

17-1. 查询可用检测数据集清单
    - 输入:无
    - 输出:全部可用检测数据集元信息列表(包含系统内置数据集与用户上传的私有数据集)
    - 触发场景:用户在启动检测前浏览可用检测数据集清单(对应 1.spec.md 功能 1.3)
    - 调用下层:PrivateConfigManager 的「查询可用检测数据集清单」;EventLogger 的「记录用户操作日志」
    - 不负责的边界:不做数据集内容的加载与解析(由 PrivateConfigManager 经 DataProcessor 完成);不做数据集元信息的 UI 渲染(由 UI 层完成)

18. 调度用户私有大模型适配器与私有数据集的上传与移除
    - 输入:操作类型(上传/移除适配器、上传/移除数据集或样本)及其所需参数(适配器文件、目标大模型标识、数据集元信息、样本条目清单、所属用户ID 等)
    - 输出:对应操作的结果(新登记大模型标识、新数据集ID、各样本ID、移除结果等)
    - 触发场景:用户管理私有大模型适配器或私有数据集(对应 1.spec.md 功能 3.1.1.2;数据集还可作为 1.spec.md 功能 1.3 的输入来源)
    - 调用下层:涉及私有资源移除前,调用 DACManager 的「判定用户对资源是否具备访问权」做前置校验;按操作类型分别调用 PrivateConfigManager 的「上传用户私有大模型适配器」「移除用户私有大模型适配器」「上传用户私有数据集」「移除用户私有数据集或其单条样本」;EventLogger 的「记录用户操作日志」
    - 不负责的边界:不做适配器入库注册的具体实现(由 PrivateConfigManager 经 LLMRegistry 完成);不做数据集/样本的具体落盘(由 PrivateConfigManager 经 DataProcessor 完成)

# 接口契约
19. 通过 StateSchedulerInterface 对外暴露上述能力,被 CLI / WebUI 调用(符合 4.模型依赖关系图.puml 中 CLI → StateScheduler 与 WebUI → StateScheduler 两条边)

不需要的:[监控系统资源使用情况,持有任何业务数据(业务数据由下层各 DB 模块持有,本模块仅持有"系统当前状态"FSM 与"执行调度上下文"等内存态),做检测算法的具体执行逻辑(由 SDPJDetector 完成),做检测结果的合规判断与统计指标计算(分别由 SDPJDetector 与 ReportManager 完成),做日志/报告/状态信息的 UI 渲染(由 UI 层完成),持久化登录会话到外部存储(登录会话由 AccountManager 在内存中维护),做密码哈希(由 UserCenter 完成),做下层错误的具体分类与重试/退避决策(分别由 LLMAdapterLib 与 LLMService 完成),越层调用抽象驱动层或基础设施层(仅可调用执行逻辑层 8 个模块的对外能力)]

依赖模块:TaskQueueManager,EventLogger,SDPJDetector,ReportManager,PrivateConfigManager,DACManager,AccountManager
调用接口:TaskQueueManagerInterface,EventLoggerInterface,SDPJDetectorInterface,ReportManagerInterface,PrivateConfigManagerInterface,DACManagerInterface,AccountManagerInterface

被依赖模块:CLI,WebUI
应实现接口:StateSchedulerInterface

技术细节:

1.系统状态管理

  1.1状态定义
    - 通过预设定好的系统状态头文件规定系统的状态

  1.2有限状态机实现
    - 通过有限状态机（FSM）的实现来跟踪系统当前所处状态
    - 根据有限状态机状态转移条件进行对应状态的转移和对应行动的执行

  1.3状态管理行为
    - 维护系统当前所处状态及状态间的转移和行为
    - 便于系统进行异常处理
    - 向用户反馈指令执行情况
    - 使用户得以了解当前系统的运行情况和检测任务的执行进度

2.调度控制模块

  2.1指令解析与转换
    - 接收用户指令，解析用户指令中的意图
    - 将用户意图指令解析，转换为实际调用指令
    - 示例：当用户选择了多个安全风险的检测类型时，为实现这一意图，转换为"一个检测任务组"
      - 检测任务组包含对不同检测数据集的检测任务
      - 调用执行逻辑层中的检测任务队列管理模块管理检测任务(并行任务是否可以通过连续入队/出队多个来实现),并调用SDPJDetector进行具体的执行

  2.2并发执行
    - 实现多个逻辑层模块的并发执行
    - 支持多个检测任务并行处理
    - 协调并调用执行逻辑层各组件执行用户指令

  2.3执行协调
    - 与系统日志与事件管理模块交互,记录和反馈系统日志
    - 向CLI/WebUI反馈指令执行情况
    - 处理系统异常和错误恢复(依赖于有限状态机的状态转移条件)
    - C-S 底层通信链路(HTTP/TCP 连接)由部署层 Web 框架在系统初始化时建立,StateScheduler 通过 StateSchedulerInterface 消费该链路,不负责链路的建立与维护
    - 敏感数据(账号密码、大模型API配置文件)的传输安全由 HTTPS/TLS 保障,不再做应用层加解密
    - 调用执行逻辑层其它模块
