EventLogger / 系统事件与日志管理模块
具体职责:

# 操作日志记录
1. 记录用户操作日志
   - 输入:操作者用户ID、操作类型、操作上下文(操作对象、关键参数等)、时间戳
   - 输出:记录结果及日志条目标识
   - 触发场景:用户触发任一业务操作时(对应 1.spec.md 功能 2.1.7.2),覆盖注册/登录/启停检测/查看或删除报告/私有配置管理/权限授予与移除等
   - 不负责的边界:不做业务操作本身的执行(仅记录);敏感信息由调用方在传入前完成脱敏

# 运行日志记录
2. 记录系统运行日志
   - 输入:事件来源模块、事件类型、事件描述、时间戳
   - 输出:记录结果及日志条目标识
   - 触发场景:系统状态机的状态转移、检测任务执行进度推进、大模型调用阶段切换等运行期事件(对应 1.spec.md 功能 2.1 / 2.1.7.2)

# 错误日志记录
3. 记录系统错误日志
   - 输入:错误来源模块、错误类型、错误描述、时间戳
   - 输出:记录结果及日志条目标识
   - 触发场景:业务执行过程中出现的异常/失败事件(对应 1.spec.md 功能 2.1.7.3),覆盖大模型 API 调用错误、数据库访问错误、权限校验失败、文件格式校验失败等

# 日志查询与检索
4. 按条件查询日志
   - 输入:过滤条件(日志类别(操作/运行/错误)、时间范围、来源模块、用户ID 等,可选)
   - 输出:匹配条件的日志条目列表
   - 触发场景:运维或用户查看系统运行状况(对应 1.spec.md 功能 2.1.7.2)
   - 不负责的边界:不做跨实例日志聚合;查询结果的展示由 UI 层承担

# 日志级别与输出目标管理
5. 设置当前生效的日志级别
   - 输入:目标日志级别(如 DEBUG / INFO / WARN / ERROR)
   - 输出:设置结果
   - 后置条件:在本模块内部维护当前生效的日志级别,低于该级别的事件不再记录/输出

6. 设置日志输出目标
   - 输入:目标输出通道(如控制台、内存缓冲、外部流等)
   - 输出:设置结果
   - 后置条件:在本模块内部维护当前生效的输出目标列表,后续日志事件按目标分发

# 日志订阅与推送
7. 订阅日志流
   - 输入:订阅者回调函数、过滤条件(可选,如日志类别、来源模块等)
   - 输出:订阅标识
   - 触发场景:上层模块(如 UI 层)需实时接收日志推送(对应 1.spec.md 功能 2.1.7.2)
   - 后置条件:后续匹配过滤条件的日志事件将回调订阅者

8. 取消日志订阅
   - 输入:订阅标识
   - 输出:取消结果
   - 触发场景:上层模块不再需要实时日志推送时

# 数据库持久化
9. 启动数据库日志写入器
   - 输入:数据库连接信息
   - 输出:启动结果
   - 后置条件:日志事件通过后台写入器异步批量持久化到 result_db
   - 触发场景:系统启动时初始化日志持久化通道

10. 清理过期日志
    - 输入:保留天数阈值
    - 输出:清理的日志条数
    - 触发场景:定期维护任务清理历史日志以释放存储空间

11. 刷新日志缓冲区
    - 输入:无
    - 输出:刷新结果
    - 触发场景:系统关闭前确保缓冲区中的日志全部落盘

# 接口契约
12. 通过 EventLoggerInterface 对外暴露上述能力,被 StateScheduler 调用(符合 4.模型依赖关系图.puml 中 StateScheduler → EventLogger 边)

不需要的:[跨实例日志聚合,敏感信息脱敏规则的制定(由调用方完成),日志内容的业务语义解析,日志的 UI 渲染展示(由 UI 层完成)]

依赖模块:result_db(数据库持久化),调用接口:无(直接使用数据库驱动)
应实现接口:EventLoggerInterface
被依赖模块:StateScheduler
技术细节:(暂定)
本模块依赖 result_db 进行日志的数据库持久化(通过 start_db_writer 启动后台写入器,将日志事件异步批量写入数据库)。其他下层持久化 / 加密 / 队列状态管理等能力由模块内部实现。
本模块通过与StateScheduler进行信息流和控制流的交换来使得本模块能够正常运行。
