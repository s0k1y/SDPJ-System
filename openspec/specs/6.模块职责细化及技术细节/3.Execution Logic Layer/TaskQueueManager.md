TaskQueueManager / 检测任务队列管理模块
具体职责:

# 任务入队
1. 接收新检测任务并加入队列尾部
   - 输入:检测任务描述(至少包含用户ID、目标被测大模型ID、静态/动态算法类型、选用的检测数据集标识等业务元信息)
   - 输出:新加入任务对应的队列任务标识
   - 后置条件:任务被追加到队列尾部,初始状态为「等待中」
   - 触发场景:用户启动一次检测时由 StateScheduler 下发任务(对应 1.spec.md 功能 1.1 / 1.2 / 1.4 / 2.1);用户连续下发多个检测任务进行批量调度(对应 1.spec.md 功能 1.4)
   - 不负责的边界:不校验任务元信息的业务合法性(由 StateScheduler 在下发前完成);不实际执行检测业务(由 SDPJDetector 完成)

# 任务出队与调度
2. 按 FIFO 顺序取出下一个待执行任务
   - 输入:无
   - 输出:队头任务及其队列任务标识
   - 后置条件:对应任务从队列头部移除,状态由「等待中」转为「进行中」
   - 触发场景:StateScheduler 选择下一个待执行任务(对应 1.spec.md 功能 1.4 / 2.1)
   - 错误情形:队列为空时返回明确错误,不阻塞调用方

3. 按 FIFO 顺序并发取出多个待执行任务
   - 输入:目标并发度上限
   - 输出:从队头起连续若干个任务及其队列任务标识
   - 后置条件:被取出的任务均从队列中移除,状态均转为「进行中」
   - 触发场景:StateScheduler 支持多任务并发调度时一次性出队多个任务(对应 1.spec.md 功能 1.4)
   - 不负责的边界:不做实际并发执行编排(由 StateScheduler 协调 SDPJDetector 完成)

# 任务状态跟踪
4. 更新单个任务的执行状态
   - 输入:队列任务标识、新状态(「等待中」/「进行中」/「已完成」/「异常中断」/「已取消」)
   - 输出:更新结果
   - 触发场景:任务进入执行态或结束时(对应 1.spec.md 功能 2.1);检测过程发生异常中断时;用户取消任务时

5. 取消指定任务
   - 输入:队列任务标识
   - 输出:取消结果
   - 后置条件:任务状态变更为「已取消」(TaskStatus.CANCELLED),若任务正在执行则通知执行协程终止
   - 触发场景:用户主动取消正在等待或执行中的检测任务(对应 1.spec.md 功能 1.4)
   - 错误情形:任务已完成或已取消时返回幂等结果

6. 查询单个任务的执行状态
   - 输入:队列任务标识
   - 输出:当前任务状态
   - 触发场景:StateScheduler 或上层 UI 查询单个任务的执行进度(对应 1.spec.md 功能 1.4 / 2.1)

7. 查询队列整体视图
   - 输入:无
   - 输出:当前队列中全部任务及其状态的列表
   - 触发场景:用户查看系统当前检测任务的排队状况(对应 1.spec.md 功能 1.4 / 2.1)

# 任务持久化
8. 持久化任务到数据库
   - 输入:任务对象
   - 输出:持久化结果
   - 触发场景:任务入队、状态更新时自动触发(通过 _persist_task / _persist_status 内部方法)
   - 调用下层:TaskPersistence(数据库持久化组件)
   - 不负责的边界:不做数据库连接池管理(由 SessionManager 完成)

# 接口契约
9. 通过 TaskQueueManagerInterface 对外暴露上述能力,被 StateScheduler 调用(符合 4.模型依赖关系图.puml 中 StateScheduler → TaskQueueManager 边)

不需要的:[支持任务暂停与恢复,实际执行检测业务(由 SDPJDetector 完成),跨任务的依赖/优先级调度(仅 FIFO),记录任务入队/出队日志(由 EventLogger 完成)]

依赖模块:TaskPersistence(任务持久化),SessionManager(数据库会话管理),调用接口:无(直接使用内部组件)
应实现接口:TaskQueueManagerInterface
被依赖模块:StateScheduler
技术细节:(暂定)
本模块内部维护一个基于 TaskPersistence 的任务队列,利用队列先进先出的机制实现多个检测任务的调度。新任务入队时通过 _persist_task 写入数据库,状态变更时通过 _persist_status 更新数据库,确保系统重启后任务不丢失。

本模块通过 SessionManager 获取数据库会话,通过 TaskPersistence 完成任务的数据库持久化。
  维护一个队列,利用队列先进先出的机制实现多个检测任务的调度,当有新的检测任务下发后自动入队,加入队尾,同时,追踪每个检测任务的执行状态。(若要实现多线程并发,这里可以考虑同时出队2-3个,就可以了)

本模块通过与StateScheduler进行信息流和控制流的交换来使得本模块能够正常运行。
