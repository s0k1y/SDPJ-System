UtilsLib / 杂项组件库模块
具体职责:

# 多模态数据处理
1. 文本到多模态数据的生成
   - 输入:文本内容、目标模态(图像/音频/视频)
   - 输出:目标模态形式的数据
   - 触发场景:DataProcessor 构造多模态注入样本数据(对应 1.spec.md 功能 1.1 / 1.2 中涉及多模态注入的检测场景)
   - 不负责的边界:不做样本业务语义解析

2. 多模态数据到文本的反向解析
   - 输入:图像/音频/视频数据
   - 输出:从中提取的文本内容
   - 触发场景:DataProcessor 解析大模型的多模态响应内容(对应 1.spec.md 功能 1.1 / 1.2)

# 多编码数据处理
3. 文本编码转换
   - 输入:源文本、目标编码形式(如 Base64、URL 编码、Unicode 转义、十六进制等)
   - 输出:目标编码下的文本
   - 触发场景:DataProcessor 构造多编码注入样本数据(对应 1.spec.md 功能 1.1 / 1.2 中涉及多编码注入的检测场景)

4. 编码文本还原
   - 输入:编码后文本、已知编码形式
   - 输出:还原后的原始文本
   - 触发场景:DataProcessor 对大模型响应中的编码内容做还原(对应 1.spec.md 功能 1.1 / 1.2)

# 文件 I/O 与格式校验
5. 文件读取
   - 输入:文件路径或文件流、读取模式(文本/二进制)
   - 输出:文件内容
   - 触发场景:DataProcessor 加载数据集文件(对应 1.spec.md 功能 1.1 / 1.2);LLMRegistry 读取用户上传的适配器文件(对应 1.spec.md 功能 3.1.1.2)

6. 文件写入
   - 输入:文件路径、待写入内容、写入模式
   - 输出:写入结果
   - 触发场景:ReportManager 通过 DataProcessor 导出检测报告文件供用户下载(对应 1.spec.md 功能 1.6)

7. 文件格式校验
   - 输入:文件内容、期望格式(如 CSV / JSON / JSONL)
   - 输出:格式合规性判定结果及关键错误信息
   - 触发场景:用户上传私有数据集文件、私有检测配置文件、私有大模型适配器文件时的格式预校验(对应 1.spec.md 功能 3.1.1.2)
   - 不负责的边界:不做内容业务语义审查(如 PoC 语义分析)

# 加密与摘要
8. 对称加密/解密
   - 输入:明文或密文、对称密钥
   - 输出:密文或明文
   - 触发场景:UserCenter 对用户密码等敏感信息进行可逆加密处理(对应 1.spec.md 功能 3.1.1.1 / 3.1.1.2)
   - 不负责的边界:不持有密钥、不做密钥生命周期管理(密钥由调用方或外部安全组件管理)

# 数据序列化
10. 结构化数据序列化
    - 输入:结构化对象、目标格式(如 JSON / YAML)
    - 输出:序列化后字符串
    - 触发场景:各调用方将结构化数据持久化到文件或跨层传递时(例如用户私有检测配置、注册中心元信息、访问控制列表的序列化)

11. 数据反序列化
    - 输入:序列化字符串、目标格式
    - 输出:结构化对象
    - 触发场景:各调用方从文件/输入中读入结构化数据时

# 接口契约
12. 通过 UtilsInterface 对外暴露上述能力,DataProcessor / LLMService / LLMRegistry / UserCenter 为调用方(符合 4.模型依赖关系图.puml 中四条 → UtilsLib 的依赖边)

不需要的:[维护业务状态与流程编排,持有或管理加解密密钥生命周期,做样本/报告/配置等数据的业务语义解析,做网络通信与协议处理,做数据库访问,做第三方大模型 API 调用]
依赖模块:无
应实现接口:UtilsInterface
被依赖模块:DataProcessor,LLMService,LLMRegistry,UserCenter
技术细节:(暂定)
核心功能是为 DataProcessor 传入的检测样本数据提供多模态/多编码处理能力,以支撑多模态注入与多编码注入的检测样本构造。
