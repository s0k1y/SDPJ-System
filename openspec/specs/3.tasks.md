```

### 1.2 依赖拓扑图

**说明**：
- Wave 0 的 T001-T005 为零依赖基础模块，T006 为独立前端骨架
- Wave 1 中 T010-T013 依赖 Wave 0 的输出
- Wave 2 中 T014-T018 依赖 Wave 1 的输出
- Wave 3 的 T019 依赖 Wave 0 的 T003-T005 和 Wave 2 的全部输出
- Wave 4 的 T020-T022 依赖 T019

**关键依赖关系**：
```

  T007, T008, T009 → T010 (DataProcessor 依赖 T001, T007, T008)

  T001, T002 → T012 (LLMRegistry 依赖 T001, T002)

  T010, T011 → T014 (SDPJDetector)

  T003, T004, T005, T014, T015, T016, T017, T018 → T019 (StateScheduler)

  T019 → T020 (CLI)

**职责清单（来自规格）:**

| 编号 | 职责 | 关键输入 | 关键输出 |
|------|------|----------|----------|
| T01 | 接收新检测任务并加入队列尾部 | 检测任务描述 | 队列任务标识 |
| T02 | 按 FIFO 顺序取出下一个待执行任务 | 无 | 队头任务 |
| T03 | 按 FIFO 顺序并发取出多个任务 | 目标并发度上限 | 连续若干个任务 |
| T04 | 更新单个任务的执行状态 | 队列任务标识、新状态 | 更新结果 |
| T05 | 查询单个任务的执行状态 | 队列任务标识 | 当前状态 |

| T07 | 接口契约暴露 | — | TaskQueueManagerInterface |

**状态机:**
```
等待中 ──(出队)──▶ 进行中 ──(完成)──▶ 已完成
                    │
                    └──(异常)──▶ 异常中断
```

**验收标准:**
- [ ] FIFO 出队顺序正确（先入先出）
- [ ] 空队列出队时返回明确错误，不阻塞调用方
- [ ] 并发出队数量不超过队列实际任务数
- [ ] 状态更新后查询结果一致
- [ ] 不支持暂停/恢复/取消（边界声明）
- [ ] 不依赖外部存储，纯内存队列

**任务子阶段:**

| 阶段 | 产出物 | 关键内容 |
|------|--------|----------|
| 1. 接口定义 | `sdpj/core/detector/interface.py` | static_detect / dynamic_detect 方法签名 |
| 2. Algorithm 1 实现 | `sdpj/core/detector/static_detector.py` | 步骤1(筛选最优PoC) + 步骤2(构造判断模型) + 步骤3(遍历生成报告) |
| 3. Algorithm 2 实现 | `sdpj/core/detector/dynamic_detector.py` | 步骤2(构造红队模型) + 步骤3(迭代变异+动态检测) |
| 4. 合规判断 | `sdpj/core/detector/judge.py` | 合规/违规判定 |
| 5. 并发处理 | — | asyncio.TaskGroup 并发处理样本 |

**Algorithm 1 核心流程:**

```
步骤1: 遍历 D_jailbreak → 筛选越狱成功 PoC → 选出最优 PoC_i
  ├── 调用 LLMInterface 发起调用
  ├── 调用 DataProcessor 加载样本/落盘结果
  └── S 为空时算法终止（无越狱风险）

步骤2: PoC_ii = PoC_i + P_role
  └── 构造虚拟化输出侧合规判断模型

步骤3: 遍历 R\{D_jailbreak} 中全部样本
  ├── 对每条 d_j:
  │   ├── Output_j = LLM(PoC_j)
  │   ├── Judge_j = LLM_judge(Output_j)
  │   └── 写入 (i, j, Output_j, Judge_j)
  └── 多模态/多编码样本通过 DataProcessor 处理
```

**Algorithm 2 核心流程:**

```
步骤1: 复用 Algorithm 1（产出静态报告）

步骤2: PoC_Attack = PoC_i + P_Attack
  └── 构造虚拟化红队攻击模型 LLM_Attack

步骤3: 对静态报告中 Judge_j=="合规" 的条目:
  └── for iteration in 1..max_iterations:
      ├── 动态变异: LLM_Attack(PoC_Current + Output_j) → 新 PoC
      ├── 攻击测试: LLM(新 PoC) → 新 Output_j
      ├── 合规判断: LLM_judge(新 Output_j) → 新 Judge_j
      ├── if Judge_j=="违规": break
      └── 更新 Report_dynamic
```

**验收标准:**
- [ ] Algorithm 1 三步全部正确执行（Mock LLM 验证）
- [ ] Algorithm 2 迭代变异+动态检测正确执行
- [ ] S 为空时算法正确终止
- [ ] 检测结果正确落盘到 ResultDB
- [ ] asyncio.TaskGroup 并发处理样本，100 个样本 <10 秒
- [ ] 合规判断输出仅限 {"合规", "违规"}
- [ ] 每次 LLM 调用完全独立（无记忆性）
- [ ] 检测类型信号量控制正确（静态→可选动态，动态→默认真值）
- [ ] 测试覆盖率 ≥90%
